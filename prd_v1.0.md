# 论证驱动型文档修订多智能体系统 PRD（MVP 快速验证版）

## Problem Statement

专业文档（如政策解读、行业研究报告、架构设计方案等）的修订是一项极度消耗智力的工作。传统的修订方式主要依赖人工重读、多源检索比对和逐字修改。用户在面对长篇文档时，常常遇到以下痛点：

1. **论证逻辑断层**：修改了某一个局部的论据或事实错误，却难以评估和传导该变动对上层分论点甚至核心主论点产生的逻辑破坏力。


2. **事实校验成本高**：需要频繁在不同系统、白名单网络和数据库间切换，难以实现“基于现有原文论断自动精准检索并交叉验证”。


3. **缺乏建设性替代方案**：当系统或人工发现某些事实论断“存疑”或“缺乏权威数据源支撑”时，传统的 AI 工具只会指出错误，无法在原文逻辑边界内精准给出“可验证的修订假设”供用户勾选和启发灵感。


4. **AI 全量重写失控**：现有的长文本大模型在接收修订指令后，往往倾向于通篇全量重写，导致原文不需要变动的完美句子、特定行文风格以及排版格式遭到破坏，产生大量“不可控的噪音”。



## Solution

本系统提供一套后端多智能体（Multi-Agent）协作修订方案，以**标准化论证树（Argumentation Tree）为全链路唯一数据主干**。系统通过单一的文档解析入口，将文章解构为具备严密父子关系的逻辑节点。

1. **乐观并行验证与开药**：系统启动后，事实验证 Agent（负责“体检”）与假设生成 Agent（负责“开药”）乐观并行。事实验证 Agent 负责查明原文事实的真伪或存疑状态；假设生成 Agent 则专门针对可能存在漏洞或缺乏支撑的节点，在严格的原文边界内生成并行的“可验证修订假设”。


2. **双轨合并流与批注门禁**：系统将“体检”与“开药”的结果合并，对存疑/错误节点无缝挂载修订假设。质量一致性校验 Agent 扮演“贴便利贴的质检员”，不中断流程，仅以标签和批注的形式标记术语冲突或逻辑溢出，最终通过人在回路（HITL）界面将所有冲突与候选假设完整呈现给用户进行勾选。


3. **段落级原子替换缝合**：在用户最终勾选并拍板后，修订回写 Agent 依据高精度的段落映射表，仅对命中修改的局部段落进行定向爆破与无缝重写拼接，新增假设则严格追加于对应段落末尾，100% 保护原文不相关的行文风格与排版格式。



## User Stories

1. 作为一名专业文档审计员，我希望系统能够自动将上传的纯文本解析为包含主论点、分论点、论据和限定条件的标准化论证树，以便于我一目了然地看清整篇文章的逻辑骨架。


2. 作为一名数据架构师，我希望“背景叙述”和“主观评价”这类非核心逻辑内容能够作为只读的影子节点挂载在论证树上，以便于在后续多智能体协作时提供完整的上下文边界，且在回写时不丢失任何原文血肉。


3. 作为一名高级分析师，我希望事实验证 Agent 能够基于 ReAct 机制自动调整检索关键词，并在白名单网络、知识库和结构化数据源中反复查找以判别某条论据的真伪，从而帮我省去手动反复检索、求证的枯燥工作。


4. 作为一名方案评审员，我希望事实验证 Agent 在发现某条关键论据“错误”或“存疑”时，能够将此结论原地落库到论证树节点状态中，以便于后续流程能精准针对该痛点展开处理。


5. 作为一名创意写作人员，我希望假设生成 Agent 能够并行启动，并专门针对那些缺乏事实支撑或存疑的观点节点，在原文的适用边界内生成多个具备验证价值的替代假设，从而为我的文档修订提供精准的改写处方。


6. 作为一名多智能体系统运维工程师，我希望“体检”与“开药”两条链路能够乐观并行运行，最大限度压榨并发算力吞吐量，提高万字长文的处理效率。


7. 作为一名项目管理者，我希望系统能够提供一个 Merge 算子，将并行的验证结果和修订假设合并为双轨流，只在事实点塌方或存疑时激活并挂载对应的替代假设，避免生成无效数据。


8. 作为一名文档质检员，我希望一致性校验 Agent 能够独立于生成逻辑，仅对更新后的论证树进行段落级和全局的一致性扫描，指出术语不一致或潜在的逻辑冲突。


9. 作为一名追求高吞吐量的系统用户，我希望一致性校验 Agent 发现冲突时不要陷入“打回重算”的无限死循环，而是单向推进并将问题作为批注标签留存，以便于整个大流水线能够顺畅、稳健地一口气跑通。


10. 作为一名最终决策人，我希望在 HITL 节点 2（修订内容确认）中清晰看到系统打上的“事实存疑”便利贴，以及并列呈现的 3 个候选修订假设，以便于我勾选出最符合我意图的那一条。


11. 作为一名关注文档风格稳定性的作者，我希望修订回写 Agent 能够利用 `paragraph_id` 和偏移量映射，仅对被我确认修改的段落进行局部定向爆破重写，其余未变更段落直接原样复制，以防止 AI 擅自润色或导致格式坍塌。


12. 作为一名合规性审查员，我希望所有新确认的假设与补充论据能够严格作为专属标识内容，追加在对应论点所在段落的末尾，以便于我能一眼审计出哪些内容是本次修订新注入的。



## Implementation Decisions

### 1. 整体智能体架构拓扑

系统采用「1中枢 + 1主干 + 双任务并行 + 1公共检索层 + 2HITL节点 + 1批注门禁 + 1原子回写」的分层编排拓扑。所有智能体通过全局调度 Agent 转发数据，无跨模块直接调用。

```
                    ┌─────────────────────┐
                    │  全局调度Agent      │  ← 中枢编排、状态管理、HITL调度、单向流控制
                    └─────────┬───────────┘
                              │
          ┌───────────────────┼───────────────────┐
          │                   │                   │
┌─────────▼─────────┐  ┌──────▼──────┐  ┌────────▼─────────┐
│ 论证结构解析Agent │  │  HITL节点组  │  │ 一致性校验Agent   │
│ （唯一数据主干）   │  │ 1.结构确认  │  │ （质量门禁·批注） │
│                   │  │ 2.内容确认  │  │                   │
└─────────┬─────────┘  └─────────────┘  └─────────┬─────────┘
          │                                       │
          └───────────────────┬───────────────────┘
                              │
                    ┌─────────▼───────────┐
                    │  双任务执行层(并行)  │
                    │  假设生成Agent      │ ──┐ 
                    │  事实验证Agent      │ ──┼─→ 运行「双轨合并算子」
                    │  影响传导Agent      │ ──┘
                    └─────────┬───────────┘
                              │
                    ┌─────────▼───────────┐
                    │  公共检索能力层     │  ← 底层公共支撑，双任务统一调用
                    │ 网络检索Agent       │
                    │ 知识库检索Agent     │
                    │ 结构化数据检索Agent │
                    └─────────┬───────────┘
                              │
                    ┌─────────▼───────────┐
                    │  修订回写Agent      │  ← 基于段落ID原子缝合拼接
                    └─────────────────────┘

```

### 2. 智能体角色详细定义（含原始职责与 MVP 硬约束）

#### 2.1 中枢层：全局调度 Agent

* **核心职责**：全链路任务编排、论证树全生命周期状态管理、消息路由、HITL 节点触发、异常兜底、任务生命周期管控。


* **输入**：上游系统传入的原始文档 + 任务配置（开启/关闭任务、修订等级、是否启用人工确认）。


* **输出**：最终修订结果 + 全链路执行日志。


* **MVP 硬约束规则**：取消复杂的分布式重试降级与跨模块挂起逻辑。当收到下游单个 Agent 的执行异常（Exception）或超时信号时，原地将目标节点的 `status` 变更为相应的错误状态，附带报错日志至节点内容中，全局调度单向向前推进，决不因单个节点波动卡死整篇长文档流水线。

#### 2.2 主干层：论证结构解析 Agent

* **核心职责**：全文唯一语义解析入口，输出标准化论证树，作为全链路唯一数据主干。


* **核心逻辑**：
1. **全文语义分区与影子节点挂载**：将内容划分为事实型论据、观点型论断、限定条件、背景叙述、主观评价 5 类。前 3 类作为核心逻辑节点纳入论证树；**后 2 类（背景叙述、主观评价）作为影子节点（只读）同样挂载在论证树上**，其状态默认为无需校验，不参与逻辑传导与事实校验，但作为完整的上下文供其他 Agent 消费，并参与最后的文本拼接。


2. **层级构建**：自顶向下识别「主论点→分论点→论据→限定条件」的从属关系，构建树状结构。


3. **元数据标注**：为每个节点绑定段落 ID（`paragraph_id`）、原文绝对偏移量（`text_span`）、论证权重。


4. **状态初始化**：所有原始节点状态标记为「未校验（`unverified`）」。





#### 2.3 任务执行层（双任务并行、可插拔）

##### 2.3.1 假设生成 Agent

* **核心职责**：基于论证树目标节点的完整上下文，生成符合原文论证框架的可验证假设。


* **核心逻辑**：直接从论证树拉取目标节点及其下属所有影子节点的完整上下文，严格在原文适用边界内生成可证伪的并行修订假设列表，自动挂载至对应节点下，节点状态标记为「待验证（`pending_verification`）」。



##### 2.3.2 事实验证 Agent

* **核心职责**：基于论证树的从属关系，调用多源检索校验论据真实性，输出校验结论。


* **MVP 局部闭环规则**：**Agent 内部采用 ReAct（Reasoning-Action）机制**。此内生 Loop **仅聚焦于“为了查清事实而自动调整检索词”**。一旦通过公共检索层查到明确的客观比对素材（不管是真是假）或触发硬上限超时，立即退出 Loop，原地写回节点状态（可信/存疑/错误），**绝对不在内部直接参与对原文论断文本的重写或修改**。

##### 2.3.3 影响传导 Agent

* **核心职责**：论据/假设的校验结果沿论证树向上传导，评估对上层论点的影响，输出修订建议。


* **核心逻辑**：按论证权重计算单条论据失效后的剩余支撑力度，低于阈值时向上逐层传递影响，输出修订建议方案（修正论据/弱化论点/补充论据）。



#### 2.4 公共检索能力层（底层公共支撑、全链路复用）

三类检索 Agent 统一接收检索请求，返回带来源标识的素材结果，同时服务于事实验证与假设验证。

* **网络检索 Agent**：仅访问预置官方白名单域名，禁止泛网搜索；检索请求脱敏，不传入文档敏感内容。


* **知识库检索 Agent**：内部非结构化知识库的语义检索，调用前自动校验用户权限，支持按文档类型、时间过滤。


* **结构化数据检索 Agent**：关系型数据库精准查询，仅执行预定义查询模板，禁止泛化 SQL 查询。



#### 2.5 门禁层：一致性校验 Agent

* **核心职责**：执行段落级校验（自洽性、边界匹配度）与全局校验（跨段落论点一致性、术语定义一致性）。


* **MVP 硬约束规则**：**完全剥离其“拒绝、打回、重新调度”的权力，规避大模型打回死循环（Rejection Loop）**。校验 Agent 仅做中立质检，发现的所有冲突、不一致和瑕疵，统一以批注便利贴（`issue_tags`）的形式注入目标节点的元数据中，流水线单向一冲到底，把问题毫无保留地展现给 HITL 2。



#### 2.6 输出层：修订回写 Agent

* **核心职责**：将最终状态论证树的所有变更，转换为自然语言修订版文档。


* **MVP 核心逻辑（局部分缝合替换）**：**坚决杜绝全量重写**。它遍历通过 HITL 2 固化后的终版论证树，对比初始段落映射表。仅针对用户最终确认勾选修改的 `paragraph_id` 调用大模型进行局部段落级重写，其余未变更段落直接从只读缓存中读取并进行流式拼接。新注入的有效假设与补充论据，统一作为专属标识内容，追加在对应论点所在段落字符串的末尾。



### 3. 统一数据标准与协议规范

#### 3.1 论证树节点数据规范（TypeScript 形状定义）

系统内全链路数据实体统一读写标准，结构定义如下：

```typescript
interface ArgumentationNode {
  node_id: string;          // 节点唯一标识，全链路通用[cite: 1]
  node_type: 'main_claim' | 'sub_claim' | 'evidence' | 'qualification' | 'background' | 'evaluation'; // 包含影子节点[cite: 1]
  parent_id: string | null; // 父节点ID，顶层节点为null[cite: 1]
  children_ids: string[];   // 子节点ID列表[cite: 1]
  paragraph_id: string;     // 所属线性段落表映射ID[cite: 1]
  text_span: { start: number; end: number }; // 对应原文起止绝对偏移量[cite: 1]
  content: string;          // 节点对应原文/生成/修订内容[cite: 1]
  argument_weight: number;  // 论证权重0-100，用于影响传导计算[cite: 1]
  boundary?: string;        // 适用边界与限定条件摘要[cite: 1]
  status: 'unverified' | 'pending_verification' | 'credible' | 'doubtful' | 'error' | 'corrected' | 'invalid'; // 状态机流转规则[cite: 1]
  evidence_list: Array<{ source: string; snippet: string }>; // 支撑/反驳素材列表[cite: 1]
  issue_tags: string[];     // 一致性校验Agent贴上的批注便利贴标签
  candidate_hypotheses?: Array<{ text: string; confidence: number }>; // 乐观并行产出的并行修订假设[cite: 1]
}

```

#### 3.2 节点状态机流转硬约束

所有节点状态流转严格遵循以下规则，非法变更直接拦截：

* 初始状态：`unverified`（未校验）


* 假设生成后：`pending_verification`（待验证）


* 检索校验后：`credible`（可信）/ `doubtful`（存疑）/ `error`（错误）


* 人工/系统修订后：`corrected`（已修正）


* 论据失效后：`invalid`（失效）



#### 3.3 双轨合并算子（Merge Operator）逻辑规则

在双任务并行完成后，调度中枢运行合并算子对结果进行排定与取交集：

* **逻辑 A**：若事实验证判定节点状态为 `credible`（原文事实正确、立得住），系统启动“以静制动”原则，自动裁剪（Drop）掉假设生成 Agent 为该节点产出的修订假设，保持原文原封不动。
* **逻辑 B**：若事实验证判定节点状态为 `error` 或 `doubtful`（原文事实点塌方或无法被现有数据源抓取支撑），系统立刻**强行激活**对应的 `candidate_hypotheses`（修订处方），将其作为候选修订方案与节点状态并列固化，完整推送至 HITL 节点 2。

### 4. 人在回路（HITL）节点定义

#### 4.1 节点1：论证结构确认（切分确认）

* **触发时机**：论证结构解析 Agent 输出初始论证树后，任务执行层启动前。


* **人工操作**：调整节点层级、合并/拆分节点、修正节点边界、标记无需处理的段落。


* **输出**：人工确认后的最终版论证树主干。（支持跳过，跳过则直接进入下一环节）。



#### 4.2 节点2：修订内容确认（双轨决策大闸）

* **触发时机**：双任务执行完成、双轨合并算子运行结束、一致性校验标签追加完毕后，最终回写前。


* **界面呈现**：用户可一眼看到被贴上 `doubtful/error` 状态及 `issue_tags` 批注便利贴的段落，并并列看到系统开出的 3 个候选修订假设。
* **人工操作**：单条勾选采纳/驳回修订假设、手动修改节点内容。


* **输出**：人工拍板确认后的终版论证树，直接驱动原子回写。

## Testing Decisions

### 1. 好的测试标准

* **外部行为验证（Black-box Testing）**：测试应聚焦于大流水线的宏观输入输出表现（“输入纯文本 -> 论证树数据流转 -> 局部分割拼接缝合后的最终文本”），绝对不能断言或测试 Agent 内部的 Prompt 结构、中间迭代次数或具体的私有类变量。
* **字节级对齐断言**：测试的核心重点在于，断言在经历了一系列复杂的并行 Agent 处理后，那些用户未勾选修改的段落，其文本内容、标点符号、换行符在输出时必须与原始输入（Byte-for-byte matching）100% 绝对一致，验证局部爆破缝合代码的强确定性。

### 2. 待测核心模块

* **ArgumentationTree Merge & Logic Test**：模拟注入 `credible` 和 `error` 的多源检索状态，断言双轨合并算子的剪枝逻辑与假设激活逻辑是否严格符合设计。
* **Paragraph-level Stitching Unit Test**：输入一段包含多个段落的标准示例文档，模拟仅确认修改中间某一个 `paragraph_id`，断言回写输出的全文中，非变更段落完全无损还原，变更段落末尾正确追加专属标识假设。

## Out of Scope

1. **多模态内容解析**：本版本仅支持纯文本（Markdown/TXT）格式的文档输入与解构，暂不支持 PDF 中的扫描图表、图片内嵌表格等复杂多模态数据的提取。


2. **通用文本润色与高级排版**：本系统不包含任何错别字纠正、修辞手法美化以及复杂的 Word/PDF 样式格式排版调整。


3. **前端交互界面渲染**：本 PRD 仅规范多智能体系统的后端调度、协作协议、合并算子逻辑与回写机制，不包含任何前端 UI 交互细节。
