# 论证驱动型文档修订多智能体系统 PRD v2.0

> 本文件为完整、独立的产品需求文档，可脱离任何其他文档单独使用。

---

## Problem Statement

专业文档（政策解读、行业研究报告、架构设计方案等）的修订是高强度智力劳动。
用户在修订长篇文档时，反复遭遇四类痛点。

1. **论证逻辑断层**：改动了某处局部论据或事实，却难以评估该变动向上传导到分论点乃至主论点时造成的逻辑破坏。
2. **事实校验成本高**：需要在多个系统、白名单网络与数据库间频繁切换，难以「基于原文论断自动检索并交叉验证」。
3. **缺乏建设性替代方案**：当某条事实论断存疑或缺乏权威数据源支撑时，通用 AI 工具只会指出错误，无法在原文逻辑边界内给出「可验证的修订假设」供用户勾选。
4. **AI 全量重写失控**：现有长文本模型接到修订指令后倾向通篇重写，破坏原文中无需变动的句子、行文风格与排版格式，产生大量不可控噪音。

---

## Solution

本系统是一套后端多智能体协作修订方案。
它以一棵**论证树**作为全链路唯一数据主干，把文档解构为具备父子从属关系的逻辑节点。
系统的核心承诺是「精准、可控、逐字保护原文」：只改用户拍板要改的地方，其余部分逐字节原样保留。

系统的工作方式可以概括为四步。

1. **确定性解构**：先用纯代码把原文无损切成有序段落并固化为不可变副本，再由解析智能体在这个只读底座上识别论证节点、构建论证树。
   原文字节从不整篇进入任何智能体的推理上下文。

2. **双线路乐观并行**：系统同时跑两条线路。
   「体检」线路对论据与论点做正向检索校验，判定其可信、存疑或错误。
   「开药」线路针对节点投机生成「可验证修订假设」，并对每条假设自身做证据检索，判定其成立、存疑或被推翻。

3. **双轨合并与批注门禁**：合并算子按一张确定性矩阵将两条线路的结果标注到同一棵树上，决定保留原文、替换、改写还是补充。
   影响传导智能体据此沿树向上评估论点是否因失去支撑而失效。
   一致性校验智能体只贴批注标签、不打回，把所有冲突与候选完整呈现给用户。

4. **人在回路与原子回写**：用户在修订确认界面逐条勾选采纳或驳回。
   回写智能体仅对被采纳改动命中的段落做局部重写或段尾追加，其余段落从只读副本逐字节拷回，100% 保护原文风格与格式。

## User Stories

1. 作为文档审计员，我希望系统自动把纯文本解析为含主论点、分论点、论据、限定条件的论证树，以便一目了然看清全文逻辑骨架。
2. 作为数据架构师，我希望「背景叙述」「主观评价」这类非核心逻辑内容作为只读影子节点挂在树上，以便提供完整上下文，且回写时不丢失任何原文血肉。
3. 作为系统集成者，我希望原始文本在解析前先被纯代码无损切成有序段落并固化为不可变副本，以便为后续任意处理提供一份逐字节可信的还原底座。
4. 作为文档校对员，我希望切分后的段落按序拼接必须逐字节等于原始输入（含空行、缩进、换行、末尾空格），以便字节级还原不依赖任何模型质量。
5. 作为审计员，我希望每个实质段落至少对应一个论证节点、影子段落对应影子节点，以便论证树的形态贴合「每段有论点、多段服务于一个上层论点」的自然结构。
6. 作为审计员，我希望一个论证节点只归属于一个段落、一个段落可含多个节点，以便回写的原子边界清晰、不产生跨段歧义。
7. 作为审计员，我希望在结构确认环节调整节点层级、合并或拆分节点、修正边界、标记无需处理的段落，并允许跳过此环节，以便在最坏情况下也绝不改动原文一个字。
8. 作为高级分析师，我希望「体检」线路基于自动调整检索词的推理机制，在白名单网络、知识库与结构化数据源中反复查找，判别某条论据的真伪，以便省去手动反复检索的枯燥工作。
9. 作为方案评审员，我希望「体检」线路在判定论据可信、存疑或错误后就地把结论落库到节点状态，以便后续流程精准针对痛点处理。
10. 作为创意写作人员，我希望「开药」线路与体检乐观并行，针对节点在原文适用边界内投机生成多条可证伪的修订假设，以便为我提供精准改写处方。
11. 作为运维工程师，我希望投机生成只覆盖论据与分论点两类节点、跳过主论点与限定条件与影子节点，以便在真并行的同时把生成算力收敛到合理范围。
12. 作为分析师，我希望每条修订假设也经过同一检索层取证，产出「成立、存疑、被推翻」三态之一，以便区分「有支撑的好药」「没查实的参考」「已被证伪该丢弃的错药」。
13. 作为分析师，我希望每条假设在生成时就被标注它与原文的语义关系（对立、递进、扩展）且一条假设只承载一种关系，以便回写通道唯一、且我能分条勾选。
14. 作为项目管理者，我希望合并算子按「原文状态 × 假设状态」的确定性矩阵裁决，只在原文塌方或存疑且假设成立时激活候选，以便避免生成无效数据。
15. 作为决策人，我希望原文可信时系统一律保持原文不动（以静制动），哪怕存在成立的扩展型假设也不擅自增补，以便最大化保护立得住的内容。
16. 作为决策人，我希望当原文可信却出现一条同样有证据、结论相反的对立假设时，系统不自动裁决，而是贴冲突标签把两者并列交我判断，以便处理数据源打架的真实情况。
17. 作为方案评审员，我希望影响传导在合并之后串行运行、且不生产任何替代文本，只做失效向上传导与复用已有假设，以便修订假设永远只有一个来源。
18. 作为审计员，我希望影响传导按论证权重计算父论点的剩余支撑率，低于阈值判失效、居中贴弱化批注，以便失效判定可解释、可复算。
19. 作为质检员，我希望一致性校验独立于生成逻辑，只对标注完成的论证树扫描一次，指出术语不一致或潜在逻辑冲突。
20. 作为追求高吞吐的用户，我希望一致性校验发现冲突时不打回重算，而是把问题作为批注标签留存、单向推进，以便整条流水线一口气稳健跑通。
21. 作为决策人，我希望在修订确认界面清晰看到被标为存疑或错误的段落、系统贴的批注标签，以及并列呈现的候选修订假设，以便勾选最符合我意图的那条。
22. 作为决策人，我希望修订确认是不可跳过的硬闸门、系统绝不在无人拍板时自动改原文，以便守住「保护原文」的底线；仅当全篇无任何待决内容时可一键通过。
23. 作为决策人，我希望我采纳某条假设后系统立即持久化这一决策，以便回写失败并重试再失败时我的决定不丢失。
24. 作为关注风格稳定的作者，我希望回写只对被我确认改动命中的段落做局部重写、其余段落逐字节原样拷回，以防 AI 擅自润色或格式坍塌。
25. 作为作者，我希望采纳的假设按其与原文的关系分流：对立则替换原句、递进则改写原句、扩展则在段尾追加，以便修订形态符合各自语义。
26. 作为合规审查员，我希望所有新注入的扩展型补充都作为带专属标识的内容追加在对应段落末尾，以便一眼审计出本次修订新增了什么。
27. 作为运维工程师，我希望编排器在任一下游智能体异常或超时时就地把目标节点置为错误状态并附日志、单向向前推进，以便单点波动绝不卡死整篇长文流水线。
28. 作为运维工程师，我希望回写是幂等操作、可安全重试，以便中断后按「已采纳未回写」的节点续跑而不重复注入。
29. 作为安全负责人，我希望网络检索只访问预置白名单域名、请求脱敏，知识库检索先校验用户权限，结构化检索只走预定义模板，以便检索过程可控合规。
30. 作为决策人，我希望界面上候选假设数量按实际生成动态呈现而非固定为某个数字，以便真实反映系统开出的处方。

## Implementation Decisions

### 1. 整体智能体架构拓扑

系统采用「1 中枢 + 1 主干 + 双线路并行 + 1 公共检索层 + 2 人在回路节点 + 1 批注门禁 + 1 原子回写」的分层编排。
所有智能体经全局调度中枢转发数据，无跨模块直接调用；流水线严格单向、绝不打回。

关键顺序一句话：
解析 →〖HITL-1〗→（体检 ∥ 开药，共用检索层）→ 双轨合并 → 影响传导 → 一致性校验 →〖HITL-2〗→ 回写。

与直觉容易搞错的三处，特别标注：
影响传导**在合并之后串行**，不在并行层（它依赖体检结论，物理上无法与体检并行）。
公共检索层是**双线路共用的旁挂支撑**，不是流水线上的一环。
只读原文段落表**旁路贯穿全程**，回写时按 `paragraph_id` 拷回，但其 bytes 永不整篇进入任何 Agent 上下文。

### 2. 双层存储

系统内存在且仅存在两处数据实体，用段落标识 `paragraph_id` 做唯一外键关联。

- **只读原文段落表（Raw Paragraph Store）**：`{ paragraph_id → 原始 bytes }` 的不可变副本。
  它是字节级还原的唯一真相源，也是修订确认界面「原文对比左栏」的数据源。
  任何智能体的推理上下文都不整篇加载它。
- **论证树**：在编排状态中流转的轻结构。
  节点只携带自身那一小段原文（作为推理输入）加 `paragraph_id` 指针，绝不存整篇原文。
  论证树可整棵进入智能体上下文而不撑爆窗口。

### 3. 段落切分（确定性、无损、纯代码）

原文进入系统后，先由**纯代码**按 Markdown 块级元素与空行边界切成有序段落（一个标题、段落、列表项、代码块各成一段）。
切分完成后校验**分区不变式**：所有段落按序拼接必须逐字节等于原始输入。
此步骤零模型参与，因此字节级还原是代码级确定的，与解析质量无关。
段落表在解析智能体介入前就已固化并通过校验。

### 4. 论证树与节点

解析智能体在只读段落底座上工作，只识别节点、挂树、回填指针，无权新建、改写、重排段落。

节点分两类。
核心逻辑节点：主论点、分论点、论据、限定条件，参与校验与逻辑传导。
影子节点：背景叙述、主观评价，只读，不参与校验与传导，但提供上下文并参与最终文本拼接。

结构约束：
一个节点只归属一个段落（`paragraph_id` 为单数），一个节点不可跨段落，一个段落可含多个节点。
跨多段展开的论点由树的父子结构消化——上层论点独立成段，其展开段各自成为子节点。

结构原则（解析器软启发式，非校验硬约束）：
每个实质段落至少对应一个节点，影子段落对应影子节点，段落主节点通过父指针归属到某个上层论点。
解析器不得为无论点的段落硬造论点。

节点数据形状（TypeScript 描述，形状为决策、非最终代码）：

```typescript
interface ArgumentationNode {
  node_id: string;
  node_type: 'main_claim' | 'sub_claim' | 'evidence' | 'qualification' | 'background' | 'evaluation';
  parent_id: string | null;
  children_ids: string[];
  paragraph_id: string;                         // 单数；节点不可跨段
  text_span: { start: number; end: number };    // 仅段内辅助定位，回写不依赖
  content: string;                              // 节点自身的原文/生成/修订文本，非整篇原文
  argument_weight: number;                      // 0-100，用于影响传导计算
  boundary?: string;
  status: 'unverified' | 'pending_verification'
        | 'credible' | 'doubtful' | 'error'
        | 'adopted' | 'corrected' | 'invalid';
  evidence_list: Array<{ source: string; snippet: string }>;
  issue_tags: string[];                         // 一致性校验贴的批注，含 'conflict' 等
  candidate_hypotheses?: Array<{
    text: string;
    relation: 'oppose' | 'advance' | 'expand';  // 一假设一关系，生成时钉定
    status: 'supported' | 'doubtful' | 'refuted';
    confidence: number;                         // 仅用于同节点内多条 supported 假设排序
  }>;
  adopted_hypothesis_id?: string;               // HITL-2 采纳后持久记录，回写幂等依据
}
```

### 5. 双线路（乐观并行）

**线路 1 · 体检**：对论据与论点做正向检索校验。
内部采用「推理—行动」循环，仅聚焦「为查清事实而自动调整检索词」。
一旦查到明确比对素材或触发超时硬上限即退出，就地写回节点状态 `credible / doubtful / error`。
绝不在内部重写或修改原文文本。

**线路 2 · 开药**：对节点投机生成假设，再对假设取证。
投机生成不依赖体检结论（二者并行），只覆盖论据与分论点两类节点，跳过主论点、限定条件与影子节点。
生成时同步标注每条假设与原文的语义关系（对立 / 递进 / 扩展），一条假设只承载一种关系；混合意图必须拆成多条。
每条假设经同一公共检索层取证，产出 `supported / doubtful / refuted`；「无假设」由空数组表达。

状态语义对称：原文 `credible/doubtful/error` ↔ 假设 `supported/doubtful/refuted`。

### 6. 公共检索能力层

三类检索智能体统一接收请求、返回带来源标识的素材，同时服务于体检与假设取证。

- **网络检索**：仅访问预置官方白名单域名，禁止泛网搜索，请求脱敏。
- **知识库检索**：内部非结构化知识库语义检索，调用前校验用户权限，支持按类型与时间过滤。
- **结构化数据检索**：关系型数据库精准查询，仅执行预定义模板，禁止泛化 SQL。

### 7. 双轨合并算子（确定性矩阵）

合并算子把两条线路的结果标注到同一棵树上（系统全程只有一棵树，此处「合并」指合并两线路的结果，不是合并两棵树）。
裁决按「原文状态 × 假设状态」的完整矩阵执行。

| 原文 \ 假设 | 成立（supported） | 存疑（doubtful） | 被推翻（refuted） | 无假设 |
|---|---|---|---|---|
| **credible** | 冲突：对立型→贴 `conflict` 并列推 HITL-2；递进/扩展型→冻结不动 | 保留原文，丢弃假设 | 保留原文，丢弃假设 | 保留原文 |
| **doubtful** | 按关系 替换/改写/补充 | 原文入 HITL-2，假设弱呈现「未证实·供参考」 | 丢弃假设，原文入 HITL-2 | 原文入 HITL-2，无候选 |
| **error** | 按关系 替换/改写/补充 | 原文入 HITL-2，假设弱呈现 | 原文与假设均错，入 HITL-2 无有效候选 | 原文错、无药，入 HITL-2 无候选 |

「成立」列的操作由假设与原文关系决定：
**对立 → 替换（replace）**、**递进 → 改写（rewrite）**、**扩展 → 补充（supplement，段尾追加带审计标识）**。

两条关键边界裁决：
原文可信且对立假设亦成立时，系统不自动裁决，贴 `conflict` 交人判。
原文可信且假设为递进/扩展型时，严格冻结、原文不动。

系统全流程**没有独立人工兜底分支**：所有节点（含无候选者）一律流入修订确认（HITL-2）这一唯一人在回路终点。

### 8. 影响传导（串行、不产文本）

影响传导在合并之后串行运行，读取标注完成的树。
它不生产任何替代文本，只做两件事：把失效沿论证权重向上传导；若某上层论点因此塌方，触发复用该节点已有的成立假设去激活。

失效判定公式（起步方案，参数可调）：
父论点剩余支撑率 = 存活直接子节点权重之和 ÷ 全部直接子节点权重之和。
低于 0.5 判 `invalid`；0.5 至 0.7 贴「弱化」批注但不失效；不低于 0.7 不受影响。
权重由解析智能体建树时按明文 rubric 赋值（带数据或引源的直接论据高分、泛泛断言低分）。

状态分工：
`error` 是体检对叶子论据的判决（论据自证其伪）。
`invalid` 是影响传导对上层论点的判决（失去支撑被拖垮）。

### 9. 一致性校验（批注门禁）

一致性校验在影响传导之后、修订确认之前扫描一次那棵标注完成的树。
它执行段落级校验（自洽性、边界匹配）与全局校验（跨段论点一致、术语定义一致）。
它**完全不具备拒绝、打回、重新调度的权力**，所有冲突与瑕疵统一以批注标签 `issue_tags` 注入节点元数据，单向推进到修订确认。
它看不到 `adopted / corrected`（那是修订确认之后才有的状态），且不因用户采纳可能引入新冲突而回炉重扫。

### 10. 人在回路

**节点 1 · 结构确认**：解析输出初始论证树后、线路启动前触发。
用户可调整层级、合并或拆分节点、修正边界、标记无需处理的段落。
**支持跳过**（跳过则直接进入下一环节，不改动原文）。

**节点 2 · 修订确认**：合并、影响传导、一致性校验全部完成后、回写前触发。
界面并列呈现被标为 `doubtful/error` 的段落原文、系统贴的 `issue_tags`（含 `conflict`）、以及候选修订假设。
用户逐条勾选采纳或驳回、可手动修改节点内容。
**此节点为不可跳过的硬闸门，系统绝不在无人拍板时自动采纳假设**；任务配置中的「是否启用人工确认」开关只作用于结构确认节点，对本节点无效。
仅当全篇所有节点均可信、无任何待决内容时，本节点呈现「无需修订」一键通过（属闸门内无待办，非跳过闸门）。

用户采纳某条假设时，节点进入 `adopted` 状态并持久记录 `adopted_hypothesis_id`。

### 11. 原子回写

回写智能体遍历修订确认后的终版树，以**段落为唯一原子单位**产出修订版文档，坚决杜绝全量重写。

- 未被任何采纳改动命中的段落：按 `paragraph_id` 从只读原文表**逐字节流式拷回**。
- 被命中的段落：按被采纳假设的关系分流处理。
  对立型 → 替换原句；递进型 → 局部改写原句；扩展型 → 在段落末尾追加带**专属审计标识**的内容。

回写按段落幂等：成功则节点置 `corrected`；失败则节点停留 `adopted` 并贴错误标签，重试时扫描「`adopted` 且未 `corrected`」的节点、读其 `adopted_hypothesis_id` 重做回写，不重复注入。

### 12. 状态机

```
unverified → pending_verification → (credible | doubtful | error)
                                         │ HITL-2 采纳
                                         ▼
                                     adopted ──回写成功──▶ corrected
                                        └────回写失败────▶ 停留 adopted（可幂等重试）

invalid：由影响传导对上层论点单独判定
```

非法状态变更一律拦截。

### 13. 全局调度中枢

职责：全链路任务编排、论证树全生命周期状态管理、消息路由、人在回路触发、异常兜底。
硬约束：取消复杂的分布式重试降级与跨模块挂起。
收到下游异常或超时时，就地将目标节点置为相应错误状态并附日志，单向向前推进，绝不因单点波动卡死整篇流水线。

## Testing Decisions

### 测试哲学

- **黑盒外部行为验证**：测试聚焦大流水线的宏观输入输出（纯文本 → 论证树流转 → 局部缝合后的终稿文本）。
  绝不断言智能体内部的提示词结构、迭代次数或私有变量。
- **字节级对齐**：核心断言是——经过一系列并行处理后，未被勾选改动的段落，其文本、标点、换行必须与原始输入逐字节完全一致。

### 测试缝合点（seam）

- **首选最高缝**：编排器边界。
  输入纯文本、注入受控的检索结果与人在回路选择，断言终稿文本。
- **三个纯函数子缝**（皆为确定性、可独立单测，优先复用而非新增）：
  1. 段落切分：文本 → 有序段落，断言分区不变式。
  2. 双轨合并算子：两线路结果 → 标注后的树，断言矩阵裁决。
  3. 段落回写：终版树 + 只读段落表 → 终稿文本，断言未变更段落字节无损、变更段落正确缝合。

### 待测核心模块

- **分区不变式测试**：任意样例文档切分后按序拼接，断言逐字节等于原文（含空行、缩进、换行、末尾空格）。
- **合并矩阵测试**：模拟注入 `credible/doubtful/error` × `supported/doubtful/refuted/无` 的组合，断言每格裁决（保留 / 替换 / 改写 / 补充 / 贴 `conflict` / 冻结）严格符合矩阵。
- **段落缝合测试**：输入多段落样例，仅确认修改中间某一段，断言终稿中非变更段落逐字节无损还原、变更段落按关系正确替换或改写或段尾追加带标识内容。
- **失效判定公式测试**：给定子节点权重与存活情况，断言父节点是否按 0.5 / 0.7 阈值置 `invalid` 或贴弱化批注。
- **回写幂等测试**：模拟回写中断后重试，断言按 `adopted` 续跑、不重复注入、最终状态收敛为 `corrected`。
- **单向不卡死测试**：模拟单个下游智能体抛异常或超时，断言该节点置错误状态、流水线仍推进至终稿。

## Out of Scope

1. **多模态内容解析**：仅支持纯文本（Markdown / TXT）输入与解构，不支持 PDF 扫描图表、图片内嵌表格等多模态提取。
2. **通用润色与高级排版**：不含错别字纠正、修辞美化、Word/PDF 样式与格式排版调整。
3. **前端交互界面渲染**：仅规范后端调度、协作协议、合并逻辑与回写机制，不含任何前端 UI 细节。
4. **采纳后二次一致性校验**：用户在修订确认中采纳的多条修订之间若互相冲突，本版不做自动二次检验，由用户在界面自行审视。
5. **可信内容的增量增强**：原文判为可信时，即便存在成立的扩展型假设也一律冻结、不擅自增补，此能力留待后续版本。
6. **无人值守全自动跑批**：修订确认为不可跳过的硬闸门，本版不提供自动采纳高置信假设的无人模式。

## Further Notes

- **失效阈值（0.5 / 0.7）与权重 rubric** 为经验起步值，须用真实文档回归调优。
- **候选假设数量** 按实际生成动态呈现，界面不固定为某个数字。
- **`conflict` 标签** 是数据源打架（原文可信且对立假设亦成立）时的专属信号，恒定交人判、系统不自动裁决。
- 系统的一切设计围绕一条不可动摇的主线：**字节级保护原文，一切改动必经人判**。
